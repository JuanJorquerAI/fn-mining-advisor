# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Surface Assessment

Phase 2 installs and configures WordPress as a headless CMS. **No new public-facing frontend UI surfaces are built in this phase.** The frontend HTML/CSS/JS shell (Phase 1) is unchanged.

The UI contract for this phase covers two narrowly scoped areas:

1. **WordPress admin copywriting** — ACF field labels and instructions that the client reads when editing content. These must be clear, professional Spanish that matches the brand's authoritative tone.
2. **CF7 form response copy** — The success/error messages returned by the Contact Form 7 submit endpoint. These are the first client-visible copy that touches the contact form interaction, and they must be ready before Phase 4 wires the frontend to them.

The existing design token contract from Phase 1 is unchanged and carries forward to all future phases.

## Copywriting Contract

Phase 2 copywriting is scoped to the WordPress admin interface (ACF field labels, ACF field instructions, CF7 form configuration) and the CF7 API response messages. No public-facing copy is introduced in this phase.

### ACF Field Group Labels (WP Admin — visible to client)

Labels must be clear Spanish that tells the client exactly what to write. Field names (machine keys) are locked in RESEARCH.md and must not change.

| Field Name | Admin Label (ES) | ACF Instruction Text |
|------------|-----------------|----------------------|
| `hero_title` | Título del héroe | Headline principal del hero. Máx. 60 caracteres. Ej: "Consultoría técnica en minería de alta precisión" |
| `hero_subtitle` | Subtítulo del héroe | Descripción breve bajo el título. 2-3 líneas técnico-comerciales. Ej: "Más de 20 años en operaciones de oro y cobre..." |
| `hero_cta_primary` | CTA primario (botón dorado) | Texto del botón principal. Verbo de acción + objeto. Ej: "Solicitar diagnóstico inicial" |
| `hero_cta_secondary` | CTA secundario (botón ghost) | Texto del botón secundario. Ej: "Ver servicios" |
| `hero_background` | Imagen de fondo del hero | Imagen profesional de minería. Mínimo 1920×1080px. JPG o WebP. |
| `pillar_N_title` | Pilar N — Título | Nombre corto del pilar de confianza. Máx. 40 caracteres. |
| `pillar_N_text` | Pilar N — Texto | Descripción del pilar. 2-3 oraciones. |
| `pillar_N_icon` | Pilar N — Icono SVG | Código SVG completo del icono. No incluir `<script>` ni estilos inline. |
| `step_N_title` | Paso N — Título | Nombre del paso metodológico. Máx. 50 caracteres. |
| `step_N_text` | Paso N — Descripción | Explicación del paso. 2-4 oraciones. |
| `service_N_title` | Servicio N — Título | Nombre del servicio. Máx. 60 caracteres. |
| `service_N_description` | Servicio N — Descripción | Descripción del servicio. 3-5 oraciones. |
| `service_N_focus` | Servicio N — Foco técnico | Etiquetas técnicas separadas por ·. Ej: "NI 43-101 · PEA · FS" |
| `service_N_icon_url` | Servicio N — URL del icono | URL absoluta del icono SVG subido a la biblioteca de medios. |
| `experience_text` | Texto de experiencia | Párrafo editorial completo sobre la trayectoria profesional. Admite HTML básico. |
| `experience_highlights` | Hitos destacados | Un hito por línea. Ej: "+20 años en minería de oro y cobre" |
| `cta_title` | Título del CTA final | Frase de cierre orientada a conversión. Máx. 70 caracteres. |
| `cta_text` | Texto del CTA final | Párrafo de apoyo al CTA. 2-3 oraciones. |
| `cta_primary` | Botón primario del CTA | Texto del botón dorado de conversión. Ej: "Solicitar diagnóstico" |
| `cta_secondary` | Botón secundario del CTA | Texto del botón de WhatsApp. Ej: "Escribir por WhatsApp" |
| `insight_category` | Categoría del insight | Categoría temática. Ej: "Metalurgia", "Sostenibilidad", "Regulación" |
| `insight_excerpt` | Extracto del insight | Resumen de 2-3 oraciones del artículo. |
| `insight_featured_image` | Imagen destacada | Imagen principal del artículo. Mínimo 1200×630px. |

### Seed Content (ACF placeholder copy — must match Phase 1 HTML)

The ACF field seed content must copy the Spanish placeholder text already present in `index.html` (Phase 1). This ensures API-hydrated content in Phase 4 matches what static HTML showed. Do not invent new copy — extract from the existing HTML.

### Destructive Actions

No destructive actions in Phase 2. WordPress admin has its own delete/trash confirmation UI (not built or modified in this phase).

## Phase 2 Specific Notes

### No New Frontend UI Surfaces

The following are explicitly OUT OF SCOPE for Phase 2 UI work:
- No new CSS files
- No new HTML sections
- No new JavaScript modules
- No changes to `index.html` or any Phase 1 file
- No custom WordPress theme (TwentyTwentyFive used as minimal active theme; never rendered publicly)

### WP Admin Is Not a Custom UI

The WordPress admin interface uses WordPress's native design system. No custom admin styling is built. The only "UI" delivered is the ACF field labels and instructions, which appear as plain text inside WP admin forms.

### Token Carry-Forward

All design tokens from `css/tokens.css` carry forward unchanged into Phase 3 (API Integration Layer) and Phase 4 (Section Integration). Any dynamically hydrated HTML elements in Phase 4 must use the same CSS custom properties already declared — no new tokens are to be introduced without updating `css/tokens.css`.
